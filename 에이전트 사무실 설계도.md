# 🏢 에이전트 사무실 — 전체 설계도 (Master Plan)

> 여러 AI 에이전트에게 **직함·역할**을 부여하고 한 프로젝트를 **협업으로 완성**시키되,
> 그 과정을 사무실 공간에서 **캐릭터가 움직이는 모습**으로 실시간 시각화하는 웹 앱.
>
> 최종 수정: 2026-07-22 · 상태: 1단계 프로토타입 완료

---

## 1. 비전과 확정 방향

| 항목 | 결정 |
|---|---|
| 성격 | **하이브리드(단계적)** — 시각화 먼저 완성 → 이후 실제 에이전트 실행 연결 |
| 비주얼 | **아이소메트릭 2.5D**, 코드로 그린 도형(외부 에셋 0) |
| 플랫폼 | **브라우저 웹 앱** |
| MVP | 움직이는 사무실 프로토타입 (4명 핵심 팀) |
| 팀 구성 | PM(김기획) · 개발자(이개발) · 디자이너(박디자) · QA(최검수) |
| 연출 수준 | 이동 + 머리 위 상태 아이콘/말풍선 |

**성공의 정의**: "지금 이 팀이 무슨 프로젝트를, 누가, 어느 단계까지 하고 있는지"를
화면만 봐도 직관적으로 이해할 수 있다. 최종적으로 그 화면이 **가짜 애니메이션이 아니라
실제 Claude 에이전트들의 작업을 반영**한다.

## 2. 핵심 설계 원칙 — "이벤트 스트림 분리"

이 프로젝트 전체를 관통하는 단 하나의 원칙. **모든 단계 설계가 여기서 나온다.**

```
 [이벤트 소스]            [번역/연출]           [월드 상태]        [렌더러]
 무슨 일이 일어났나   →   어떻게 보여줄까    →   지금 상태     →   화면에 그림
 ─────────────       ─────────────       ──────────     ──────────
 1단계: 스크립트      Choreographer       World Model     Isometric
 3단계: 실제 에이전트  (프런트엔드)         (agents/tasks)  Renderer
```

- **이벤트 소스**만 단계마다 교체된다 (스크립트 디렉터 → WebSocket → 실제 에이전트 런타임).
- **번역·월드·렌더러는 한 번 만들면 그대로 재사용**된다.
- 그래서 "가짜로 움직이기 → 진짜로 일하기"가 프런트엔드 재작성 없이 가능하다.

### 두 층위의 이벤트

| 층위 | 예시 | 누가 만드나 | 목적 |
|---|---|---|---|
| **의미(Semantic) 이벤트** | `task.assigned`, `agent.tool_call`, `task.handoff`, `agent.output` | 에이전트 런타임(백엔드) | "실제로 무슨 일이 일어났는가" |
| **연출(Choreography) 명령** | `move`, `status`, `say`, `meeting.start` | Choreographer(프런트) | "그걸 화면에서 어떻게 보여줄까" |

> 백엔드는 **의미 이벤트만** 보낸다. "개발자가 어느 책상으로 걸어가야 하는지"는 몰라도 된다.
> 그 번역은 전적으로 프런트엔드 Choreographer의 몫. 이 경계가 흐려지면 하이브리드가 깨진다.

## 4. 데이터 모델 & 이벤트 프로토콜 (계약)

> 이 스펙이 **프런트/백엔드 사이의 유일한 계약**. 여기만 지키면 양쪽을 독립적으로 개발 가능.

### 4.1 엔티티

```ts
type AgentId = 'pm' | 'dev' | 'designer' | 'qa';
type Status  = 'working' | 'walking' | 'meeting' | 'idle' | 'blocked'; // blocked = 막힘/반려
type Stage   = '기획' | '개발' | '디자인' | 'QA' | '완료';

interface Agent {
  id: AgentId;
  name: string;          // 김기획
  role: string;          // PM · 프로젝트 매니저
  color: string;         // 캐릭터 색
  pos: [number, number]; // 타일 좌표(부동)
  status: Status;
  speech: { text: string; until: number } | null;
}

interface Task {
  id: number;
  name: string;          // 로그인 기능
  stage: Stage;
  ownerId?: AgentId;
  artifacts?: string[];  // 3단계: 실제 산출물(파일 경로/텍스트)
}
```

### 4.2 이벤트 스펙 (버전 있는 JSON)

모든 이벤트는 `{ v, type, ts, ...payload }` 형태. `v`는 프로토콜 버전.

**의미 이벤트 (백엔드 → 프런트)**
```jsonc
{ "v":1, "type":"task.create",   "ts":0, "taskId":1, "name":"로그인 기능", "stage":"기획" }
{ "v":1, "type":"task.assigned", "ts":0, "taskId":1, "agentId":"dev" }
{ "v":1, "type":"agent.thinking","ts":0, "agentId":"dev", "summary":"인증 흐름 설계 중" }
{ "v":1, "type":"agent.tool_call","ts":0,"agentId":"dev", "tool":"write_file", "target":"auth.ts" }
{ "v":1, "type":"agent.output",  "ts":0, "agentId":"dev", "taskId":1, "artifact":"auth.ts" }
{ "v":1, "type":"task.handoff",  "ts":0, "taskId":1, "from":"dev", "to":"qa" }
{ "v":1, "type":"task.advance",  "ts":0, "taskId":1, "stage":"QA" }
{ "v":1, "type":"meeting.start", "ts":0, "topic":"데일리 스탠드업" }
{ "v":1, "type":"meeting.end",   "ts":0 }
```

**연출 명령 (Choreographer가 내부 생성 — 백엔드는 몰라도 됨)**
```jsonc
{ "type":"move",   "agentId":"dev", "to":[5,3] }
{ "type":"status", "agentId":"dev", "status":"working" }
{ "type":"say",    "agentId":"dev", "text":"코딩 중... ⌨️", "duration":3 }
```

### 4.3 번역 규칙 (의미 → 연출) 예시

| 의미 이벤트 | Choreographer 동작 |
|---|---|
| `task.assigned(dev)` | 개발자 → 자기 책상으로 `move`, 도착 시 `status: working`, `say:"맡았어요!"` |
| `agent.tool_call` | 해당 에이전트 `say`로 도구/파일명 표시, `status: working` |
| `task.handoff(dev→qa)` | 개발자가 QA 책상으로 걸어가 `say:"리뷰 부탁해요"` 후 자리 복귀 |
| `meeting.start` | 전원 회의실 좌석으로 `move` → `status: meeting` |
| `agent.thinking` | 머리 위 💭 아이콘 + summary 말풍선 |

### ✅ 3단계 — 실제 에이전트 백엔드 연결 (핵심, 완료)

- **목표**: 화면의 움직임이 **실제 Claude 에이전트의 작업**을 반영.
- **기술 결정(설계 시점 재판단, 이후 사용자 요청으로 재선택)**
  - **인증: Claude Code 구독 사용** → **Claude Agent SDK**(`@anthropic-ai/claude-agent-sdk`)로 구현.
    이유: 채팅/Claude Code 구독은 Messages API(종량제 API 키)와 별도 시스템이라, 구독으로 쓰려면
    Claude Code를 라이브러리화한 Agent SDK가 정답. 이미 로그인된 `~/.claude` 자격증명을 자동 사용
    (`ANTHROPIC_API_KEY` **미설정** 시). 별도 API 키·결제 불필요.
    - 에이전트는 내장 파일 도구(Write/Read/Edit)로 샌드박스(cwd)에 작업하고, 그 **도구 호출**을 의미 이벤트로 번역.
    - `permissionMode: 'bypassPermissions'`(무인 서버 자동 승인), `disallowedTools: ['Bash']`(셸 차단).
  - **WebSocket 대신 SSE(Server-Sent Events)**. 이벤트는 서버→클라 단방향이라 SSE가 정확히 맞고,
    브라우저 내장 `EventSource`로 전송 계층 **의존성 0**.
  - **비용 안전장치**: 기본 mock 모드(무료). `OFFICE_LIVE=1` 일 때만 실제 실행(구독 소모).
  - **6역할 확장 + 고도화**(2026-07-22): PM(김기획)·아키텍트(정설계)·개발(이개발)·디자이너(박디자)·QA(최검수)·테크라이터(윤문서).
    각 역할에 전문 시스템 프롬프트 + 산출물 지침(`guide`) + 역할별 모델.
    파이프라인: 기획→설계→디자인→개발→QA→문서→완료.
  - **역할별 모델**: PM·아키텍트·개발 = `claude-opus-4-8`, 디자이너·QA·라이터 = `claude-sonnet-5`
    (`roles.mjs`의 `model` 필드). `OFFICE_MODEL` 설정 시 전원 오버라이드. 구독 등급 허용 범위여야 함.
  - 대안(참고): 종량제를 원하면 Claude API 키 + Tool Runner 경로도 가능(이전 커밋 방식).
  - **모델별 사용량 표시**: 각 에이전트 실행의 `result` 토큰/비용을 `agent.usage` 이벤트로 방출 →
    프런트가 모델별로 누적해 사이드바에 표시(입력/출력 토큰, 비용은 있을 때만). 구독은 비용 0일 수 있어 토큰 위주.
- **한 일**
  1. ✔ `server/` 백엔드 (Node ESM, 의존성 0 HTTP+SSE). `GET /events`(SSE), `POST /project`, `GET /health`.
  2. ✔ 역할 정의(`roles.mjs`): PM·개발·디자인·QA 시스템 프롬프트.
  3. ✔ 실제 에이전트(`agents.mjs`): Tool Runner로 PM 분해(`create_task`), 개발/디자인 산출물(`write_file`),
     QA 판정(`report`). 도구 run 안에서 의미 이벤트 방출 + 샌드박스 저장.
  4. ✔ 오케스트레이터(`orchestrator.mjs`): 기획→개발→디자인→QA→완료 상태 머신 + 반려 시 1회 재작업.
  5. ✔ 목 오케스트레이터(`mock.mjs`): 키 없이 파이프라인 검증용.
  6. ✔ 샌드박스(`workspace.mjs`): `server/workspace/<projectId>/` 경로 탈출 차단.
  7. ✔ 프런트 `SSESource` + 주제 입력 UI. `?live` 쿼리로 소스 전환(`main.js` 한 줄 갈아끼우기).
- **완료 기준**: ✔ SSE 파이프라인을 헤드리스로 검증 — mock 30초 실행 시 이벤트 34개,
  9개 타입 전부 발생, 두 태스크 모두 '완료' 도달. 실제(live) 경로는 Tool Runner API로 코드 완성
  (키·`npm install` 후 실행; 이 환경에선 미실행).
- **실행**: `cd server && npm install` → `OFFICE_LIVE=1 ANTHROPIC_API_KEY=... node src/index.mjs`,
  프런트는 `node web/serve.mjs` 후 **http://localhost:5173/?live** 접속.
- **남은 리스크**: 실제 LLM 지연/비용은 effort·모델로 조정; 산출물 열람 UI는 4단계.

### 🎨 4단계 — 완성도 & 확장 (선택)

- 팀 규모 확장(6~8명), 방/층 추가, 카메라 이동·줌.
- 도구별 세밀 애니메이션, 감정 표현, 사운드.
- 프로젝트 히스토리 리플레이(이벤트 로그 재생 → **1단계 스크립트 소스가 곧 리플레이어**).
- 여러 프로젝트 동시 진행, 관전 공유 링크.

## 7. 기술 스택 결정

| 영역 | 선택 | 이유 |
|---|---|---|
| 렌더링 | Canvas 2D (직접 아이소메트릭 투영) | 의존성 0, 캐릭터 소수라 성능 충분 |
| 프런트 빌드 | Vite + TypeScript (2단계~) | 모듈화·타입 안정성·빠른 dev 서버 |
| 통신 | WebSocket (실시간 push) | 이벤트 스트림에 최적 (대안: SSE) |
| 백엔드 | Node.js + TypeScript | 프런트와 언어 통일, Agent SDK 지원 |
| 에이전트 | Claude Agent SDK | 도구 사용·에이전트 루프 내장 |

> 1단계는 의존성 0 단일 파일 유지. 2단계에서 Vite 도입 시에도 산출물은 정적 웹으로 배포 가능.

## 8. 결정된 사항 (2026-07-22 확정)

| # | 질문 | 결정 | 설계 영향 |
|---|---|---|---|
| 1 | 프로젝트 입력 방식 | **자연어 주제 입력** | 3단계에 PM 에이전트의 "요구사항 분해" 단계 필수. 프런트에 주제 입력 UI 추가 |
| 2 | 산출물 저장 위치 | **샌드박스 폴더** (`workspace/`) | 백엔드가 프로젝트 내 격리 폴더에만 파일 쓰기. 도구 권한을 이 경로로 제한 |
| 3 | 비용/속도 제어 | **이벤트 완급 조절** (큐 + 재생 속도) | Choreographer 앞단에 이벤트 큐. LLM의 불규칙한 타이밍을 화면 속도로 평탄화 |
| 4 | 실패 연출 | **풍부한 연출** | 막힘/반려 시 좌절 애니메이션·재작업 왕복·색 변화. 3~4단계 연출 작업량 증가 |
| 5 | 배포 범위 | **나중에 결정** | 로컬 기준으로 개발. API 키는 로컬 `.env`에만(커밋 금지). 호스팅은 3단계 이후 판단 |

### 결정에 따른 구조 반영

- **입력 UI**: 프런트 상단에 "프로젝트 주제" 입력창 → `project.start(topic)` 이벤트로 백엔드 전달.
- **샌드박스**: `server/workspace/<projectId>/` 아래에만 파일 생성. 에이전트 파일 도구의 루트를 여기로 고정(경로 탈출 차단).
- **이벤트 큐**: `web/src/eventQueue.ts` — 수신 이벤트를 버퍼링해 일정 페이스로 Choreographer에 공급. `agent.tool_call` 폭주 시에도 캐릭터가 순간이동하지 않음.
- **실패 상태 추가**: `Status`에 `blocked`(막힘) 추가, 태스크에 `rejected`/`retry` 이벤트. 반려 시 QA→담당자 왕복 연출.
- **시크릿**: `.env`(API 키)는 반드시 `.gitignore`에 포함. 커밋 금지 규칙 준수.

## 부록. 현재 코드와 이 설계의 매핑

| 설계 개념 | 현재 `index.html` 위치 |
